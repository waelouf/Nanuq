The Nanuq v2.1 key protection design:

IKeyProtector
├── DpapiKeyProtector     (Windows — unchanged)
├── KeychainKeyProtector  (macOS)
├── LibSecretKeyProtector (Linux)
└── EnvelopeKeyProtector  (Docker/K8s — one env var)

Each credential is encrypted v2$<keyId>$<iv>$<ct>$<tag>.

App refuses to boot without a valid protector. No silent plaintext fallback.

## LinkedIn

```
I posted the Nanuq v2.1 roadmap today.

The headline change: removing the Windows DPAPI dependency.

Nanuq v2.0 ships secure credential storage (AES-256 encrypted), but the key derivation uses Windows DPAPI — which means the encryption either doesn't work or silently degrades on Linux and macOS. That's not acceptable for a tool designed for Docker-based self-hosting.

v2.1 introduces an IKeyProtector abstraction with four implementations:
- DpapiKeyProtector (Windows — unchanged)  
- KeychainKeyProtector (macOS, SecKeychain)
- LibSecretKeyProtector (Linux, libsecret)
- EnvelopeKeyProtector (cross-platform — reads NANUQ_MASTER_KEY from env or ~/.config/nanuq/master.key)

The EnvelopeKeyProtector is the key one for Docker users. Inject one secret, get proper credential encryption on any Linux container. The app refuses to start without a valid protector — no silent fallback to insecure storage.

We're also shipping a migration command for existing credentials and a GHSA advisory for the v1 derivation weakness.

Full roadmap in the GitHub Discussion: [link]

If you've been watching Nanuq and waiting for Linux/macOS support — it's coming.

GitHub: github.com/waelouf/Nanuq

#opensource #dotnet #selfhosted #devops #linux
```

## GitHub Discussion Reply Templates

When community members respond, use these templates:

**Reply to "When will this ship?":**
```
No committed date yet — development is actively underway. The IKeyProtector + EnvelopeKeyProtector work (which unlocks Linux self-hosting immediately) is the highest priority first task. Follow the repository for updates.
```

**Reply to "Can I contribute?":**
```
Yes! The macOS KeychainKeyProtector and Linux LibSecretKeyProtector are both good contribution targets — the interface is simple (Protect/Unprotect), the hard part is the P/Invoke layer into platform-specific APIs. If you have experience with that, open an issue and we can work through the approach together.
```

**Reply to "Why not use [alternative]?":**
```
Happy to discuss alternatives. The EnvelopeKeyProtector approach was chosen because it works in containerized environments without OS-level keyring access. For bare-metal Linux/macOS deployments, LibSecret/Keychain are better choices. The abstraction supports both.
```
