If you've looked at Nanuq and liked what you saw but couldn't use it because you run Linux or macOS — v2.1 is for you.

## The Problem with v2.0

Nanuq v2.0 and v2.0.1 ship with AES-256 encrypted credential storage. That's the good news. The bad news: the key derivation is tied to Windows DPAPI (`ProtectedData.Protect`). This means:

- The app runs fine on Windows
- On Linux or macOS, credential encryption either silently falls back to an insecure path or fails entirely
- You can't use Nanuq in a Docker container on a Linux host with secure credential storage

This is a significant limitation for a dev tool that's primarily deployed via Docker Compose.

## What v2.1 Changes

v2.1 introduces a proper key protection abstraction:

```
IKeyProtector (interface)
├── DpapiKeyProtector    (Windows — existing behavior, unchanged)
├── KeychainKeyProtector (macOS — SecKeychain API)
├── LibSecretKeyProtector (Linux — libsecret / GNOME Keyring)
└── EnvelopeKeyProtector  (cross-platform — master key from env var or file)
```

The `AesGcmCredentialService` replaces the current AES-CBC implementation and uses envelope encryption: each credential is encrypted with a per-record key, wrapped by whichever key protector is active for the platform.

**For Docker/Kubernetes users specifically:** The `EnvelopeKeyProtector` reads a master key from `NANUQ_MASTER_KEY` (environment variable) or `~/.config/nanuq/master.key` (file, mode 600). This means you can run Nanuq in a Linux container with proper credential encryption by injecting one secret.

The app will **refuse to start** if no acceptable key protector is configured — no silent insecure fallback.

## Migration Path

v2.1 ships with a migration command that reads existing v1 AES-CBC ciphertexts and re-encrypts them with the new AES-GCM envelope format. There will also be a transparent read path for the v1 format during the transition window.

## What's Not Changing

The UI, the API, and all 5 platform integrations (Kafka, Redis, RabbitMQ, AWS SQS/SNS, Azure Service Bus) stay the same. v2.1 is entirely focused on the credential storage layer.

## Self-Hosting on Linux/macOS — Now Possible

With v2.1, the Docker Compose deployment path becomes fully cross-platform:

```bash
git clone https://github.com/waelouf/Nanuq
cd Nanuq
# Set your master key
export NANUQ_MASTER_KEY=$(openssl rand -base64 32)
docker compose up -d
```

That's the target experience. One environment variable, cross-platform secure storage.

## Timeline

Development is underway. The v2.1 implementation is sequenced as:

1. `IKeyProtector` interface + `EnvelopeKeyProtector` + `AesGcmCredentialService` (critical path — unlocks Linux self-hosting immediately)
2. macOS `KeychainKeyProtector` (parallel)
3. Linux `LibSecretKeyProtector` (parallel)
4. Migration command + cross-version fixtures
5. K8s deployment validation + docs update
6. GHSA advisory for the v1 derivation weakness

No committed release date yet, but the GitHub Discussion has the full technical scope.
